SECTION 1

System Overview

Architecture, technology stack, and high-level capabilities of SwiftShip

SwiftShip is a full-stack, role-based logistics management system designed for Nepal-based courier companies. It covers the complete delivery lifecycle — from order creation by vendors to last-mile delivery by riders — with integrated payments, inventory, and reporting.

🖥️ Frontend

React 18 + Vite + TypeScript
Tailwind CSS, shadcn/ui
TanStack Query (data fetching)
Port 5000

⚙️ Backend API

Node.js + Express 5
REST API with JWT Auth
Drizzle ORM + PostgreSQL
Port 8080

🗄️ Database

PostgreSQL (Neon/Replit)
Drizzle schema migrations
Audit logging
Real-time notifications

SECTION 2

Role & Access Overview

User roles, login credentials, and portal routing

🔴 Admin

Full system access. Can manage users, settings, audit logs, bank accounts, all data. Only role that can delete entities or change system configuration.

All modules

🔵 Manager

Operations control. Manages orders, riders, vendors, stations, payments. Cannot access user management, audit logs, or system settings.

Operations

🟠 Staff

Limited read access. Can view orders and use support tickets. Cannot create, assign, or approve orders. Read-only role for tracking staff.

View only

🟣 Vendor

Business client portal. Creates and tracks their own orders, manages COD payments, views performance reports, manages bank accounts.

Own data

🟢 Rider

Field delivery agent portal. Views assigned orders, updates pickup/delivery status, manages physical inventory, tracks commission earnings.

Assigned orders

ℹ️ Branch Manager / Accountant

These functional roles map to Manager (branch operations) and Admin/Manager (payment approval) within the current RBAC system. No separate portal — same interface with appropriate permissions.

Login & Role Routing Flow

👑

Admin Portal

Route prefix: /admin/*  ·  Role: admin (and some sections: manager, staff)

The Admin portal provides complete control over all system data, users, settings, and financial operations. The Admin is the only role that can access user management, audit logs, system branding settings, and corporate bank accounts.

Sidebar Modules

Dashboard

KPI overview: total orders, revenue, COD summary, active riders, vendor stats, performance charts

Orders

View/search all orders across all vendors. Filter by status, station, date. Create new orders.

Assign Orders

Batch assign unassigned orders to riders. Select multiple orders, pick a rider, assign with one click.

Duplicate Review

Review orders flagged as potential duplicates (same customer phone + address). Confirm or dismiss flags.

Vendors

Create/edit vendor accounts, set commission rates, view vendor performance summary.

Riders

Manage rider profiles, assign coverage stations, view commission earnings, backfill commissions.

Stations

Manage delivery hubs/branches. Each order and rider is linked to a station.

Stock Inventory

Manage physical product stock. Stock-in/out movements, rider inventory assignment and returns.

Payments

Process vendor payment requests (approve → release). Manage rider commission payouts. View all transactions.

Users *Admin only

Create/edit/delete all platform users. Reset passwords. View linked vendor/rider profiles.

Bank Accounts *Admin only

Manage corporate bank accounts used for vendor payouts.

Audit Logs *Admin only

Immutable log of all create/update/delete actions by all users with timestamp and IP.

Settings *Admin only

System branding (logo, name, tagline), general configuration, support contact management.

Support Tickets

View and respond to all tickets from vendors and riders. Real-time message thread per ticket.

Admin Dashboard KPIs

Order Metrics
  • Total orders today / this month
  • Orders by status (pending, in-transit, delivered, returned)
  • Delivery success rate %
Financial
  • Total COD collected
  • Amount pending release
  • Amount already released
  • Vendor payment queue
Operations
  • Active riders count
  • Top performing riders
  • Station performance breakdown
  • Open support tickets

Vendor Portal

Route prefix: /vendor/*  ·  Role: vendor  ·  Data: own orders/payments only

Vendors are business clients who ship products via SwiftShip. They can create orders, track deliveries, view COD (Cash on Delivery) earnings, request payment settlements, and analyse their performance. All data is strictly filtered to their own account.
Dashboard

Personal KPIs: order counts by status, COD summary (pending/released), today's delivery activity, recent comments

My Orders

View all own orders with full filter/search. Track each order status in real-time.

Create Order

Single order creation form: customer details, pickup address, package info, COD amount.

Bulk Order Upload

Upload Excel file or use grid to create multiple orders at once. Preview before submit.

Payments — COD Payment

View released (completed) COD payments. S.No, Transfer ID, Date, Amount, Bank. Download PDF statement.

Payments — Payment Request

Submit new payment withdrawal requests. Track status (pending → approved → released).

Bank Accounts

Manage bank/wallet accounts for receiving payouts. Add multiple accounts.

My Reports

Business analytics: delivery rates, COD trends, order volume charts, top products.

Today's Comments

Daily log of rider comments on own orders — customer feedback, failed delivery notes.

Stock Inventory

View own product stock levels managed by the logistics company.

Vendor Payment Workflow

1
Order Delivered → COD Collected

When rider marks order as "delivered", COD amount is recorded against the vendor's account. Status: pending release.

2
Vendor Requests Payment

Vendor clicks "Request Payment", selects bank account, enters amount, submits. Status: pending.

3
Admin/Manager Approves

Admin reviews request, approves with amount confirmation. Status changes to approved. Vendor notified.

4
Admin Marks as Released

After bank transfer, admin marks as released, adds Transfer ID and payment date. Status: released.

5
Vendor Downloads PDF Receipt

Released payment appears in "COD Payment" tab. Vendor can download PDF with full order breakdown.

Rider Portal

Route prefix: /rider/*  ·  Role: rider  ·  Data: assigned orders only

Riders are field delivery agents. They see only their assigned orders, can update delivery status at each stage, manage the physical inventory in their possession, and track commission earnings from completed deliveries.
Dashboard

Today's orders, pending pickups, deliveries completed this month, commission earned, follow-up reminders

Orders

List of all orders assigned to this rider. Filter by status. Tap to view full detail and update status.

Order Detail

Customer info, package details, delivery address map link, COD amount, status history, add comment.

My Inventory

Physical stock items currently held by the rider — assigned by admin. Track quantity, product, date assigned.

Payments — Received

Released commission payments. Transfer ID, date, amount, order count. Download PDF commission statement.

Payments — Requests

All submitted commission withdrawal requests with status (pending/released/rejected) and admin remarks.

Payments — Bank Accounts

Manage bank/wallet accounts for commission payouts. Add multiple accounts.

Support Tickets

Create and respond to support tickets — report issues, damaged packages, customer disputes.

Order Status Update Flow (Rider)

Commission PDF Statement

When a rider's payment is released, they can download a PDF Commission Statement showing: rider name, reference ID, payment date, bank account, and a full breakdown table of all orders in that payment (order code, customer name, product, commission amount, total).

SECTION 10

Key Data Models

Core database tables, fields, and relationships

orders

id, orderCode (unique), vendorId, riderId, stationId, customerName, customerPhone, deliveryAddress, productName, codAmount, deliveryCharge, paymentReleaseStatus, paymentRequestId, isDuplicate, status, createdAt

users

id, email, passwordHash, name, role (admin/manager/staff/vendor/rider), stationId, isActive, createdAt

riders

id, userId, name, phone, stationId, coverageArea, commissionRate, status, createdAt

payment_requests

id, vendorId, bankAccountId, requestedAmount, approvedAmount, status (pending/approved/released/rejected), referenceId, paymentDate, walletMethod, adminNote, createdAt

order_status_history

id, orderId, status, changedBy, note, createdAt — Immutable log of every status change on an order

rider_commissions

id, riderId, orderId, amount, status (pending/released), paymentRequestId, createdAt — Auto-created on each delivery

notifications

id, userId, title, message, type, relatedId, isRead, createdAt — In-app notifications for all events

audit_logs

id, userId, action, entity, entityId, previousData (JSON), newData (JSON), ipAddress, createdAt — Admin-only full audit trail

System Summary

SwiftShip — Key capabilities and architectural decisions

Security
  • JWT-based stateless auth
  • Role guards on all API routes
  • Immutable audit log trail
  • Data isolation per vendor/rider
  • Password hashing (bcrypt)
Scalability
  • Stateless REST API
  • Paginated order queries
  • Role-based data filtering at API layer
  • Drizzle ORM — typed SQL
  • pnpm monorepo structure
User Experience
  • Real-time in-app notifications
  • Bulk order CSV/Excel upload
  • One-click PDF receipt download
  • Duplicate order auto-detection
  • Mobile-responsive UI
